## 二、🔴 全局红线（绝对不可违反）

> **以下红线来自多个仓库的实际约束，违反任何一个都会导致系统故障。**
> 修改任何代码/配置前，必须逐条对照检查。

### 代码红线

| # | 红线 | 所在位置 | 违反后果 |
|---|------|----------|----------|
| R1 | `parseMaterialsTree()` 中 `parts.length === 4` | Dashboard/app.js | Materials 目录结构解析失效，学生列表为空 |
| R2 | `CONFIG.materials.branch` 必须为 `master` | Dashboard/app.js | 无法拉取备课材料 |
| R3 | `CONFIG.calendar.branch` 必须为 `main` | Dashboard/app.js | 无法拉取教学日历 |
| R4 | `CONFIG.profiles.branch` 必须为 `main` | Dashboard/app.js | 无法拉取学生档案 |
| R5 | `fetchFile()` 中 Contents API 的 base64 必须用 `TextDecoder('utf-8')` 解码 | Dashboard/app.js | 中文文件内容乱码或加载失败 |

### Nginx 红线

| # | 红线 | 违反后果 |
|---|------|----------|
| N1 | 不得添加 `gzip on` | JS 文件经 gzip 后浏览器解压失败，全站白屏 |
| N2 | 不得修改 `/gh-api/` 的 `proxy_pass` 地址 | GitHub API 代理失效 |
| N3 | 不得修改 `/gh-raw/` 的 `proxy_pass` 地址 | 文件下载代理失效 |
| N4 | `/gh-api/` 必须携带 `Authorization` header | 私有仓库（Student-Profiles）返回 404 |

### 文件结构红线

| # | 红线 | 所在仓库 | 违反后果 |
|---|------|----------|----------|
| F1 | Materials 目录必须严格 4 层：`学生名/角色/周次/文件名`（旧）或 `年级/作业/周次/文件名`（新作业） | Teaching-Materials | 网站解析失效 |
| F2 | 周次必须两位纯数字：`08`（不是 `第8周`） | Teaching-Materials | 网站解析失效 |
| F3 | 角色层只能是 `学生` 或 `教师` | Teaching-Materials | 网站解析失效 |
| F4 | 不可在周次和文件名之间增加子文件夹 | Teaching-Materials | 网站解析失效 |
| F5 | `parseMaterialsTree()` 依赖 `parts.length === 4` 判断，不可改动目录层级 | Teaching-Materials | 作业/学生文件全部无法解析 |

### 作业出题人标注规范

- 作业文件（`年级/作业/周次/周X-科目.md`）头部可加元数据：`> ✍️ 出题人：姓名（AI辅助）`
- Dashboard 通过 `extractAuthor()` 异步解析，仅在作业文件展示，周末讲义不展示
- 无标注默认为教师出题

### 库依赖红线

| # | 红线 | 违反后果 |
|---|------|----------|
| L1 | 所有第三方库必须放在本地 `lib/` 目录 | 国内 CDN 不稳定，改回 CDN 会导致样式/功能丢失 |
| L2 | 不得删除或重命名 `lib/` 目录下的任何文件 | 页面功能异常 |

### GitHub API 操作红线

| # | 红线 | 违反后果 |
|---|------|----------|
| A1 | **不得用 Contents API（`PUT /repos/:owner/:repo/contents/:path`）推送含中文内容的文件** | API 的 base64 编码对中文处理不稳定，文件可能变为 0 字节（2026-04-03 实际发生：Teaching-System/AGENTS.md 被 ee02867 写入 0 字节） |
| A2 | 推送文件**必须**用 `git clone + 本地编辑 + git commit + git push` | 确保文件内容完整，尤其是中文文件名和中文内容 |

## 三、备课流程（严格按顺序执行）

### 步骤 1：读取学生档案

从 `Student-Profiles` 仓库读取：
- `students.json` — 所有学生基本信息（姓名、年级、科目、上课时段、当前周次）
- `profiles/{学生名}.md` — 详细档案（当前进度、薄弱点、上次反馈、学习风格、教学策略）

> 如无法访问私有仓库，请要求教师手动提供学生信息。

### 步骤 2：读取教学日历

从 `Teaching-Calendar` 仓库读取：
- `校历总览.md` — 学期起始日期、假期安排、考试时间
- `src/{学段}/{年级}/{科目}/{学期}.md` — 各科详细教学进度（**这是唯一数据源，不要读 schedule.json**）

确认学校当前周次和教学进度。

### 步骤 3：确定本周教学重点

对比学生进度 vs 学校进度：

| 情况 | 应对策略 |
|------|----------|
| 跟上 | 按学校进度走，巩固 + 适当抢进度 |
| 落后 | 优先补薄弱点，少量抢进度 |
| 超前 | 拓展拔高 + 真题训练 |

### 步骤 4：生成全部材料

> ⚠️ 生成材料前，**必须先读取 `Teaching-Materials/备课规范.md`**，了解学生版/教师版的详细格式要求。

每科每节课只产出 **2 份文档**，按规范写入 `Teaching-Materials/{学生名}/`：

```
{学生名}/
├── 学生/{周次}/
│   └── {编号}-{科目}-{内容}（学生版）.md     ← 学案（打印给学生）
└── 教师/{周次}/
    └── {编号}-{科目}-{内容}（教师版）.md     ← 时间轴操作手册
```

| 文档类型 | 定位 | 说明 |
|----------|------|------|
| **学生版（学案）** | 好的题目 + 要点总结 | 打印给学生做题，课后带走当复习资料，**不含答案** |
| **教师版（操作手册）** | 以时间为轴的操作手册 | 题目+答案+解析+逐字稿+分支判断全部融入时间轴，老师只需这一份 |

> 📌 **标杆文件**：`杨紫泠/教师/08/01-语文-期中备考冲刺（教师版）.md`

### 步骤 5：更新学生档案

如果本次课有值得记录的情况（测验结果、薄弱点发现、家长反馈等），更新 `Student-Profiles/profiles/{学生名}.md` 和 `students.json` 的 `current_week`。

### 步骤 6：输出打印清单

给教师一个清晰的清单：
- 哪些文件需要打印（精确文件名）
- 课表概览（时间 - 科目 - 内容）

## 七、📅 周规划流程（每周一执行）

> **这是 AI 智能体每周一必须引导教师完成的流程。不可跳过。**

### 前置动作

1. **自动读取教学日历** — 读取 `Teaching-Calendar/src/` 下对应年级+科目的进度文件，列出本周学校进度
2. **自动读取学生档案** — 读取所有学生的 `profiles/{学生名}.md`，汇总薄弱点和遗留问题

### 周规划 Checklist

```
□ 本周有没有考试/特殊事件？
□ 每个课段的科目和教学目标确认了吗？
  → 先列方案，等老师确认，再动手
□ 上周遗留问题处理了吗？
□ 学生档案里有足够信息支撑备课吗？
  → 不够就问老师，一个一个来
□ 材料生成优先级排好了吗？
  → 按时间紧迫程度排序
```

### 关键原则

- **先出方案再干活** — 不要在老师确认前生成材料
- **信息不够就问** — 不要猜测学生情况，问老师
- **一个问题带出下一个** — 不要一次问一堆，聚焦当前最需要的
- **老师确认后才动手** — "可以开始生成了吗？" → 确认后再生成

## 九、🔄 文档同步规则

> **这是给 AI 智能体的强制性规则。任何接手本项目的 AI 都必须遵守。**

### 规则 1：修改代码 → 同步文档

任何对代码、配置、目录结构、数据格式的修改，**必须同步更新所有相关 AGENTS.md**。
具体来说：

| 你改了什么 | 必须更新的文档 |
|-----------|---------------|
| app.js 中的数据结构或解析逻辑 | Teaching-Dashboard/AGENTS.md |
| app.js 中的 CONFIG 或红线 | Teaching-Dashboard/AGENTS.md + 本文件（红线清单） |
| Teaching-Materials 目录规范 | Teaching-Materials/AGENTS.md + 本文件（红线清单） |
| Teaching-Calendar 数据源 | Teaching-Calendar/AGENTS.md + 本文件（步骤2） |
| Nginx 配置 | Teaching-Dashboard/AGENTS.md + 本文件（红线清单） |
| Student-Profiles 的字段结构 | Student-Profiles/AGENTS.md（如有） + 本文件（步骤1） |

### 规则 2：更新文档 → 追加变更日志

每次修改后，**必须在本文件"变更日志"章节追加一条记录**，格式：
```
[日期] [改动仓库] 改动摘要 + 影响范围
```

### 规则 3：接手项目时 → 先读日志

任何新 AI 接手本项目时，**必须先读完以下内容**，按顺序：

1. 本文件（Teaching-System/AGENTS.md）— 全局规则 + 红线 + 变更日志
2. 各仓库 AGENTS.md — 各自的技术细节
3. 代码实际实现 — 以代码为准，文档如有出入以代码为准

### 规则 4：发现不一致 → 立即修正

如果发现代码和文档不一致，**以代码为准，立即更新文档**，并在变更日志中记录。

### 规则 5：规范/命名变更 → 全量扫描

> ⚠️ **这是 2026-04-04 踩坑后新增的规则。**

修改任何规范、命名、数据源、仓库可见性后，**必须全量扫描所有仓库的所有文件**，确认旧引用全部清除。

**扫描方法：**
1. `cd /tmp && rm -rf *_repo && git clone` 每个仓库
2. 用 `grep -rn "旧关键词" /tmp/*_repo/` 全量搜索
3. 逐条判断是否需要修改（变更日志中的历史记录不需要改）
4. 修改后重新扫描，确认清零

**最容易漏的地方：**
- 多个仓库可能存在同名文件（如 `START.md` 同时存在于 System 和 Dashboard）
- 代码中的字符串匹配逻辑（如 `app.js` 的 `getFileIcon()`）
- 错误示例中的命名（如"❌ 黄涵松/学生/08/原卷/..."）
- 科目名称对照表

## 七、🚨 踩坑记录（历次事故与教训）

> **任何接手本项目的智能体必须先读完本章节。** 这些都是实际发生过的错误，不要重复踩坑。

### 事故 1：GitHub API 推送中文文件变空（2026-04-03）

**发生了什么：** 通过 GitHub Contents API（`PUT /repos/:owner/:repo/contents/:path`）推送 `Teaching-System/AGENTS.md`，文件变为 0 字节（commit `ee02867`），导致系统总规则书完全丢失。

**根本原因：** API 的 base64 编码对中文内容处理不稳定，写入时内容丢失但 API 返回了 200 成功状态码，没有报错。

**预防措施（已写入红线 A1/A2）：**
- **绝对不要**用 Contents API 推送含中文的文件内容
- 所有文件推送必须走 `git clone → 本地编辑 → git commit → git push`
- 已在仓库中的文件，也不要用 API 读取后原样写回（中文内容可能在编解码中损坏）

---

### 事故 2：规范变更只改了部分文件，导致系统内引用大面积不一致（2026-04-03 → 2026-04-04 修复）

**发生了什么：** 2026-04-03 把材料格式从 4 文件体系（原卷/解析/教案/要点）改为 2 文件体系（学生版/教师版），但只改了 `Teaching-Materials` 仓库的核心文件，漏掉了以下文件中的旧命名引用：

| 仓库 | 文件 | 旧引用数量 |
|------|------|-----------|
| Teaching-System | `START.md` | 7 处 |
| Teaching-System | `AGENTS.md` | 多处 |
| Teaching-Dashboard | `START.md` | 10 处 |
| Teaching-Dashboard | `AGENTS.md` | 2 处 |
| Teaching-Dashboard | `app.js` | 3 处 |
| Teaching-Materials | `AGENTS.md` | 1 处 |

直到 2026-04-04 逐一扫描才发现并修复。

**根本原因：** 规范变更没有做全量扫描，以为改了核心文件就完了。

**预防措施（已写入规则 5）：**
- 规范变更后必须用 `grep -rn` 全量扫描所有仓库
- 同名文件可能存在于多个仓库（如 START.md 在 System 和 Dashboard 各有一份）
- 扫描脚本保存在 `/tmp/full_scan.py`，可直接运行

### 事故 3：仓库可见性变更后文档未同步（2026-04-04 发现并修复）

**发生了什么：** Teaching-Dashboard 仓库从公开改为私有，但 `Teaching-System/AGENTS.md` 和 `Teaching-Dashboard/AGENTS.md` 中仍写着"公开"。

**根本原因：** 仓库可见性变更是直接在 GitHub 网页上操作的，没有同步更新文档。

**预防措施：**
- 在 GitHub 上修改仓库设置（可见性、分支名、仓库名）后，**立即**更新所有引用该仓库的文档
- 当前各仓库正确可见性见下方"仓库现状快照"
